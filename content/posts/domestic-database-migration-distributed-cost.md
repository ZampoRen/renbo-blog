一次“国产替代”项目刚启动，后端同学通常先问两件事：驱动还用不用换？原来的 CRUD 能不能跑？

这两个问题都该问，但它们太靠前了。MySQL 协议能连上、`SELECT` 能执行，只能证明你进了门。真正会在后面改掉日常习惯的，是另外几件事：数据不一定还在同一台机器上；同样叫 `REPEATABLE READ`，语义未必还是 InnoDB 那套；一条 `ALTER TABLE` 也不能只问“锁不锁表”。

JDBC/DSN 地址只是第一处修改。接下来，数据库会要求你对数据放在哪、事务跨不跨节点、DDL 在什么窗口执行负责。

这里不讨论谁排第一，也不把 OceanBase、TiDB、PolarDB、达梦、金仓贴成互斥的架构标签。产品线、版本和部署形态不同，答案会不同。下面说的是一个 MySQL 开发者进入评估和 POC 时，最该先改掉的五个默认动作。

## 连接串大概率能复用，但连接层不能只改 host

以 MySQL 兼容模式或 MySQL 兼容引擎为目标时，很多应用确实可以继续使用现有的 MySQL 客户端生态。TiDB 官方明确支持 MySQL 协议，MySQL client 和常见 GUI 工具可用于连接；OceanBase 的文档也提供 Connector/J、Connector/C 等连接组件。对应用来说，这通常意味着 ORM 不必从零换一套。

但“驱动没换”不等于“连接层没工作”。至少有四件事要在 POC 里跑一遍：

- 入口变了没有。 你连的是单个地址、代理地址、读写端点，还是云上的集群地址？故障切换后，已有连接什么时候失效、新连接会去哪里，这比 URL 长什么样重要。

- 认证和 TLS 变了没有。 用户、证书、加密参数、网络白名单和 VPC/DNS 往往不是 MySQL 原样搬过去的。

- 连接池的上限还是不是原来的含义。 池子不一定要重写，但 `maxOpenConns`、空闲连接、获取连接超时、重试策略都要跟着目标集群和代理的并发模型重新压测。把单库时代的连接数直接乘节点数，是很常见也很危险的误算。

- 异常是不是还长得一样。 SQLSTATE、错误码、连接中断时机、事务提交失败后的可重试性，都会影响应用的重试代码。尤其是写操作，先确认幂等，再谈自动重试。

所以连接层的交付物不该是一张“驱动版本已升级”的截图，而是一份故障回归：切换入口、短暂断链、超时提交、连接池耗尽时，业务到底报什么、重试几次、会不会重复写。

## CRUD 大多能跑；你依赖的 MySQL 细节未必能跑

普通的建表、增删改查，是兼容工作的起点，不是终点。TiDB 的官方兼容性文档就是一个很直接的提醒：它支持 MySQL 协议和常用 5.7/8.0 功能，但不支持存储过程和函数、触发器、事件、UDF、XA SQL 等一批能力。

这件事不该被理解成“某个产品缺功能”。它更像一面镜子：你需要把项目里所有不在 CRUD 范围内的依赖揪出来。

库存扣减是否藏在 trigger 里？夜间结算是否靠 event？有没有服务把 XA 当作现成能力？有没有下游只会读 MySQL binlog？ORM 生成的分页 SQL、`ON DUPLICATE KEY`、JSON 函数、字符集和排序规则，在目标版本里有没有同样的边界？

`LIMIT`、`JOIN` 也不是只验证“语法能过”就够。你还要看执行计划：连接条件能不能减少数据移动，过滤是不是在该发生的位置发生，热点查询是否把流量集中到少数数据分区。单库里一条 `JOIN` 的坏计划，常常只是慢；数据分布之后，它可能同时变成慢、贵，还让某个节点变热。

迁移前，先别去要一张“兼容率”。从慢日志、SQL 审计、代码仓库和任务调度里抽出真实语句，分成三类：直接通过、需要改写、必须重做。第三类通常不多，却最容易拖住上线。

## AUTO_INCREMENT 不是消失了；主键不再只是表结构细节

“分布式库没有自增 ID”是个过时的二分法。很多产品仍支持自增列，但它的分配方式、连续性和跨节点表现不应该靠 MySQL 的直觉下注。

TiDB 的官方文档明确写到：自增值全局唯一，但默认情况下不保证在多个 TiDB Server 之间严格连续；它提供兼容模式来处理特定需求。这个例子足够说明问题：如果业务把 ID 连续当成排序、分页游标、对账编号甚至用户可见编号，迁库前就该拆开这些责任。全局唯一、时间有序、业务可读、严格连续，从来不是同一个需求。

分布式表真正要先定的是分区键。它决定一笔订单、订单项、库存记录能不能落在相近的位置；也决定扩容时数据怎样迁移、热点会不会扎进一个分区。选分区键时，别从“哪个字段基数高”开始背答案，先把最重的事务和查询拿出来：

- 同一笔业务事务里哪些表总是一起访问？

- 最高频的写入会不会集中到一个租户、一个商家、一个商品或一个时间窗口？

- 报表和回查是否经常跨这个键扫很大范围？

你希望尽量让高频事务在本地完成，而不是每次扣库存、写订单、记流水都拉多个节点参加。OceanBase 的官方调优文档也直接建议降低跨服务器分布式事务的比例。分区键不是 DBA 在建表末尾补的参数，它是应用数据模型的一部分。

## Online DDL 还在，别再用“会不会锁表”一句话验收

MySQL 时代，给大表加索引时，大家会先盯着 `ALGORITHM`、`LOCK` 和业务低峰。这个习惯应该保留，但问题得问得更具体。

目标数据库是否支持在线 DDL，取决于产品、具体版本、部署形态和操作类型。同样是“加索引”，有的操作可以在线执行，有的仍会占用更强的元数据锁或产生明显资源竞争；在分区表上，还要确认 DDL 的作用范围、后台任务、失败后的状态和回滚路径。不能因为文档里出现了 Online DDL，就把整类操作当成无感。

发布前至少把下面四项写进变更单：

- 这条 DDL 在目标版本的确切支持等级；

- 对读、写和元数据操作分别有什么限制；

- 预计持续多久，资源高峰时会影响哪些节点或分区；

- 中断、超时或失败后，表和索引处于什么状态，怎么恢复。

最可靠的验证仍然是：拿接近生产的数据量和并发，在预发环境实跑。DDL 不是 SQL 文本审核能验出来的能力；它是一次带业务流量的运行实验。

## RC、RR 名字可以一样，事务语义不能靠名字继承

这部分最容易被忽略，因为代码里通常已经有 `BEGIN`、`COMMIT`、`SELECT ... FOR UPDATE`，看上去没有任何变化。

以 TiDB 为例，官方文档说明它以 `REPEATABLE-READ` 名称提供兼容，但实现的是 Snapshot Isolation，并且与 ANSI 的 Repeatable Read、也与 MySQL 的 Repeatable Read 存在差别；同时可以按会话设置 `READ-COMMITTED`。这不是让你记住某家产品的术语，而是在提醒：隔离级别的名字不是行为合同。

你应该把关键业务写成并发测试，而不是只执行 `SHOW VARIABLES`：

- 两笔事务同时扣同一份库存，第二笔是等待、失败还是覆盖？

- 一个事务先查再更新，期间另一笔修改了范围内的数据，业务不变量还能不能成立？

- `FOR UPDATE` 在当前事务模式下锁住了什么？

- 提交阶段出现冲突或网络抖动，应用能不能识别“可安全重试”和“必须人工对账”的区别？

分布式事务可以用。只是当一致性范围跨到多个数据位置，协调和网络路径就进来了。OceanBase 的文档把跨服务器事务比例直接列为性能调优对象；TiDB 的事务实现也依赖分布式协调。高频链路应该把真正必须跨分区的部分压到最少，并为冲突、超时和重试保留可观察性。

## 什么时候才该换：不是“国产替代”，是 MySQL 已经扛不住你的边界

话说得更直一点：慢 SQL 没分析、索引没补、长事务没治理、连接池没上限、备份恢复没演练，先换成分布式数据库，只会把这些基础问题放进更复杂的运行模型里。

真正值得进入分布式 SQL POC 的信号，是 MySQL 的边界已经成为持续的工程成本：容量或写入增长无法再靠合理拆分和扩容承接；现有分库分表把路由、跨库事务和运维压到了应用团队；故障恢复与可用性目标要求多节点协同，而团队也愿意为数据分布、热点治理和容量均衡持续投入。

如果你的第一矛盾是云上托管和既有 MySQL 生态的延续，就先按具体的 PolarDB 引擎、版本和云网络边界评估，不要把 PolarDB 与 PolarDB-X 混成一个答案。如果第一矛盾是存量兼容和可控替换，也别因为“分布式更先进”就把项目推向不必要的复杂度。

一个可上线的 POC，至少应该交出五份结果：连接与故障回归、真实 SQL 的兼容清单、分区键与热点压测、关键 DDL 的演练记录、关键事务的并发语义测试。五份都能过，才轮得到比较性能和报价。

MySQL 兼容解决的是你能不能进门。换库以后，数据和事务是否还待在同一个地方，才决定你要不要为它付下一笔账。
